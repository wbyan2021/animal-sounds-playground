# 声音博物馆 · 完整方案

> 一个社区共建的开源声音库，让孩子在家就能听到大自然的声音。
> 通过 GitHub 维护，任何人都能贡献声音，前端零改代码即可更新内容。

---

## 一、目标

| 维度 | 内容 |
|------|------|
| 核心使命 | 让 1-6 岁儿童在家通过真实声音认识自然与世界 |
| 维护方式 | GitHub 仓库，社区 PR 共建 |
| 设计原则 | 数据与代码分离；加声音零改代码；贡献门槛最低化 |
| 不做的事 | 不做管理后台；不考虑商业化；不引入后端服务 |

---

## 二、整体架构

```
社区贡献者
    │
    ├─ 提 Issue："我想加鲸鱼的声音"
    │
    └─ 提 PR：上传 MP3 + meta.json
           │
           ▼
    GitHub Actions 自动校验
    （命名规范 / meta 格式 / 音频规格 / 版权字段）
           │
           ▼
    人工 Review → 合并到 main
           │
           ▼
    build-manifest.js 自动扫描
    生成 manifest.json
           │
           ▼
    前端 fetch manifest.json
    动态渲染分类 + 卡片
           │
           ▼
    孩子点击 → 播放声音
```

**核心思路：前端永远只读 manifest.json。贡献者只需加文件 + 填 meta.json，脚本自动生成索引，前端自动更新。**

### 4.2 单条声音 meta.json（每个声音一个）

放在 `data/sounds/{category}/{name}/meta.json`：

```json
{
  "id": "animals.dog",
  "category": "animals",
  "subcategory": "pet",
  "name_zh": "狗",
  "name_en": "Dog",
  "emoji": "🐶",
  "description": "人类最忠实的朋友",
  "sounds": [
    { "file": "audio/dog_bark_01.mp3", "duration": 2.1 },
    { "file": "audio/dog_bark_02.mp3", "duration": 1.8 },
    { "file": "audio/dog_growl_01.mp3", "duration": 3.0 }
  ],
  "tags": ["宠物", "家庭", "汪"],
  "license": "CC0-1.0",
  "source": "https://freesound.org/xxx",
  "contributor": "github_username",
  "added_at": "2026-06-17"
}
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 格式 `{category}.{name}`，全局唯一 |
| `category` | 是 | 对应 categories.json 中的分类 id |
| `subcategory` | 否 | 子分类 id |
| `name_zh` | 是 | 中文名 |
| `name_en` | 是 | 英文名（也是目录名） |
| `emoji` | 是 | 用于前端显示（暂用 emoji，未来可换图片） |
| `description` | 否 | 一句话描述 |
| `sounds` | 是 | 音频文件数组，至少 1 条 |
| `sounds[].file` | 是 | 相对 meta.json 的路径 |
| `sounds[].duration` | 否 | 时长（秒），脚本可自动计算 |
| `tags` | 否 | 搜索标签 |
| `license` | 是 | 必须是 CC0 或可商用授权 |
| `source` | 是 | 音频来源 URL |
| `contributor` | 是 | GitHub 用户名 |
| `added_at` | 是 | 添加日期，脚本自动填充 |

## 五、构建脚本 build-manifest.js

### 职责

1. 递归扫描 `data/sounds/` 目录
2. 找到每个 `meta.json`，读取内容
3. 将音频文件路径补全为仓库根目录相对路径
4. 合并所有声音数据 + categories.json → `manifest.json`
5. 自动计算版本号（日期 + 当日序号）
6. 输出统计信息

### 运行方式

```bash
# 本地运行
node scripts/build-manifest.js

# CI 自动运行（合并到 main 后触发）
# 见第六节 GitHub Actions 配置
```

### 输出示例

```
✓ 扫描完成
  分类：4 个
  声音条目：37 条
  音频文件：97 个
  总体积：13.2 MB
✓ manifest.json 已生成
  版本号：2026.06.24.1
```

### 校验脚本 validate.js

在 PR 提交时由 GitHub Actions 调用，检查：

1. **目录命名**：必须是小写英文 + 下划线，如 `green_frog`
2. **meta.json 格式**：必填字段是否齐全、类型是否正确
3. **音频文件**：是否为 MP3、是否 < 200KB、是否 1-10 秒
4. **id 唯一性**：不能和已有声音重复
5. **版权字段**：license 必须是 CC0 或可商用授权，source 不能为空
6. **emoji 存在**：至少有一个 emoji

## 六、GitHub 自动化

### 6.1 PR 校验（validate-pr.yml）

**触发时机**：每次 Pull Request

**执行内容**：
```yaml
name: Validate PR
on:
  pull_request:
    paths:
      - 'data/sounds/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: node scripts/validate.js
      - run: node scripts/build-manifest.js --dry-run
```

**效果**：PR 里如果文件命名不对、meta.json 格式错误、音频不达标，Actions 报红，自动阻止合并。

### 6.2 自动构建（build-manifest.yml）

**触发时机**：PR 合并到 main 分支

**执行内容**：
```yaml
name: Build Manifest
on:
  push:
    branches: [main]
    paths:
      - 'data/sounds/**'
      - 'data/categories.json'

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: node scripts/build-manifest.js
      - run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add data/manifest.json
          git commit -m "auto: rebuild manifest.json" || echo "no changes"
          git push
```

**效果**：合并 PR 后，manifest.json 自动更新并提交回仓库。不需要人工跑脚本。

## 八、质量控制

### 8.1 自动校验（机器）

| 检查项 | 规则 |
|--------|------|
| 目录命名 | 小写英文 + 下划线 |
| meta.json | 必填字段齐全、类型正确 |
| 音频格式 | MP3 |
| 音频大小 | < 200KB |
| 音频时长 | 1-10 秒 |
| id 唯一 | 不与已有条目冲突 |
| 版权字段 | license + source 不为空 |

### 8.2 人工 Review（维护者）

| 检查项 | 说明 |
|--------|------|
| 音质 | 清晰、无杂音、非电子合成 |
| 真实性 | 是真实声音，不是 AI 生成的 |
| 响度 | 不会突然太大吓到孩子 |
| 分类 | 放对了分类 |
| 版权 | source 链接可访问、授权确认无误 |
| 适合儿童 | 内容适合 1-6 岁儿童 |

### 8.3 音频统一规范

| 项 | 标准 |
|----|------|
| 格式 | MP3 |
| 采样率 | 44100 Hz |
| 比特率 | 128 kbps |
| 时长 | 1-10 秒（自然音可到 30 秒） |
| 响度 | -16 LUFS（归一化，避免忽大忽小） |
| 体积 | 单条 < 200KB |

## 九、从 V1 迁移

### 9.1 迁移内容

| V1 位置 | 迁移到 | 方式 |
|---------|--------|------|
| `v1版本/sounds/*.mp3`（97 个） | `data/sounds/animals/{name}/audio/` | 按动物归类 |
| `index.html` 里的 `ANIMALS` 数组 | 每个动物的 `meta.json` | 逐条提取 |
| `index.html` 里的 `CATEGORIES` 数组 | `data/categories.json` | 手动整理 |
| `index.html` 里的 `REAL_SOUNDS` | meta.json 的 `sounds` 数组 | 合并到元数据 |

### 9.2 迁移步骤

```
Step 1: 创建仓库目录结构
  mkdir -p data/sounds/animals/{dog,cat,cow,...}/audio

Step 2: 复制音频文件
  把 v1版本/sounds/dog1a.mp3 → data/sounds/animals/dog/audio/dog_01.mp3
  （按命名规范重命名）

Step 3: 生成 meta.json
  为每个动物创建 meta.json，从 ANIMALS 数组提取 emoji/名称/分类

Step 4: 运行构建脚本
  node scripts/build-manifest.js

Step 5: 验证
  检查 manifest.json 是否包含全部 37 种动物
```

### 9.3 迁移后的收益

- 加声音：新建文件夹 + 写 meta.json，不用改代码
- 加分类：改 categories.json，不用改代码
- 换图片：改 meta.json 的 emoji 字段，不用改代码
- 多人协作：各自 PR，不冲突

## 十、前端改造

### 10.1 改造目标

把 V1 的 `index.html` 从「硬编码数据」改成「fetch manifest.json 动态渲染」。

### 10.2 核心改动

```javascript
// ===== V1 现在是这样 =====
const CATEGORIES = [ ... ];     // 硬编码
const ANIMALS = [ ... ];        // 硬编码
const REAL_SOUNDS = { ... };    // 硬编码

// ===== 改造后是这样 =====
async function loadData() {
  const res = await fetch('data/manifest.json');
  const manifest = await res.json();
  // 后续所有渲染逻辑从 manifest 取数据
  return manifest;
}
```

### 10.3 改动范围

| 模块 | 改动 |
|------|------|
| 数据加载 | 新增 fetch manifest.json |
| 分类渲染 | 从 manifest.categories 动态生成 tab |
| 卡片渲染 | 从 manifest.sounds 动态生成卡片 |
| 声音播放 | 从 sound.sounds[].file 路径播放 |
| 预加载逻辑 | 遍历 manifest.sounds 获取所有音频路径 |
| 其余 UI / 动画 / 音量 / Service Worker | 不变 |

### 10.4 向后兼容

前端同时支持两种模式：
1. **在线模式**：fetch manifest.json（默认）
2. **离线模式**：Service Worker 缓存 manifest + 音频

## 十三、关键决策记录

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 仓库结构 | 单仓库（data + web） | PR 一次搞定，未来可拆 |
| 图片方案 | emoji | 零版权、加载快、儿童友好 |
| 音频格式 | MP3 | 兼容性最好 |
| 部署方式 | GitHub Pages | 免费、自动 |
| 管理方式 | Git + 脚本，不做后台 | 降低维护成本 |
| 版权协议 | CC0（素材）+ MIT（代码） | 最干净，可商用 |
| 版本号 | 日期 + 序号（如 2026.06.24.1） | 直观，可对比 |

---

## 十四、风险与应对

| 风险 | 应对 |
|------|------|
| 版权纠纷 | 强制要求 source + license 字段，PR 模板勾选确认 |
| 音质参差 | 自动校验格式 + 人工 review 音质 |
| 响度不一致 | 规范要求 -16 LUFS，未来脚本可自动归一化 |
| 社区冷启动 | 先自己填 50+ 条自然声音，做出样子再推广 |
| 仓库体积膨胀 | 单条 < 200KB 限制，100 条才 20MB，GitHub 完全扛得住 |

## 附录：快速开始

```bash
# 1. clone 仓库
git clone https://github.com/wbyan2021/sound-museum.git
cd sound-museum

# 2. 添加一个新声音（例如雨声）
mkdir -p data/sounds/nature/rain/audio
cp ~/Downloads/rain.mp3 data/sounds/nature/rain/audio/rain_01.mp3

# 3. 创建 meta.json
cat > data/sounds/nature/rain/meta.json << 'EOF'
{
  "id": "nature.rain",
  "category": "nature",
  "subcategory": "weather",
  "name_zh": "雨",
  "name_en": "Rain",
  "emoji": "🌧️",
  "description": "淅淅沥沥的雨声",
  "sounds": [
    { "file": "audio/rain_01.mp3" }
  ],
  "tags": ["天气", "水"],
  "license": "CC0-1.0",
  "source": "https://freesound.org/xxx",
  "contributor": "wbyan2021",
  "added_at": "2026-06-24"
}
EOF

# 4. 本地校验
node scripts/validate.js

# 5. 生成索引
node scripts/build-manifest.js

# 6. 提交 PR
git checkout -b add-rain
git add .
git commit -m "Add rain sound"
git push origin add-rain
# 然后在 GitHub 上创建 PR
```
